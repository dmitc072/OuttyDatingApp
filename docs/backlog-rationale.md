# Product Backlog — Ordering Rationale

## How the Backlog is Ordered

The product backlog is ordered using the following priorities:

1. **Technical dependencies** – Foundational features are completed before features that rely on them.
2. **User value** – Must Have features are prioritized ahead of Should Have and Could Have features using the MoSCoW prioritization method.
3. **Risk reduction** – More technically complex features are scheduled early enough to identify and address issues before later development.

This approach ensures the team delivers a functional product increment each sprint while minimizing technical risk.

---

## Ordering Rationale

The backlog begins with user authentication and profile creation because these capabilities are required before users can access the application's core features. Once users have created a profile and selected their adventure preferences, the team can implement matching functionality. After matching is complete, messaging features are added so matched users can communicate.

---

## Summary

The backlog is ordered to:

- Complete foundational functionality first.
- Deliver the highest-value features early.
- Reduce technical risk before implementing advanced functionality.
- Defer optional features until the core product is complete.

This ordering supports an incremental development process and allows the team to deliver working software at the end of each sprint.
